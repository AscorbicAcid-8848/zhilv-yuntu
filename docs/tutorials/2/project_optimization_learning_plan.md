## 3. RAG 优化路线

### 3.1 RAG 可观测

目标：先看清楚当前 RAG 召回了什么。

建议新增：

- `backend/scripts/debug_rag_retrieval.py`

这个脚本输入：

- 目的地
- 用户偏好
- 备注
- `top_k`

输出：

- 实际检索 query
- top-k 召回片段
- 每个片段来源文件
- 每个片段相似度分数或排序位置

意义：

- 不改主业务链路，风险低
- 能直接帮助你理解当前 RAG 效果
- 后续 Query Rewrite、Rerank、Compression 都可以基于它验证

面试表达：

> 我给 RAG 链路补了调试脚本，可以直接观察用户请求对应的 top-k 召回片段，避免 RAG 变成黑盒。

---

### 3.2 RAG 小型评估集

目标：不要只凭感觉判断 RAG 好不好。

建议新增：

- `backend/tests/fixtures/rag_eval_cases.json`

样例结构可以类似：

```json
[
  {
    "name": "大理轻松看日落",
    "destination": "大理",
    "preferences": ["自然风景", "拍照", "美食", "轻松"],
    "notes": "不想太早起床，希望安排一个适合看日落的地点。",
    "expected_keywords": ["洱海", "双廊", "喜洲", "日落", "拍照"]
  }
]
```

先准备 10 条左右即可。

评估维度：

- 是否召回目标城市相关内容
- 是否召回用户偏好相关内容
- 是否召回关键景点或主题标签
- 是否存在明显无关片段

面试表达：

> 我没有只看单次生成效果，而是构造了一组旅行查询样例，用来观察 RAG 召回结果是否覆盖目的地、偏好和关键场景。

---

### 3.3 Query Rewrite / Query Correction

目标：让用户口语化输入变成更适合检索的 query。

当前问题：

- 用户输入可能很口语化
- 偏好和备注可能比较散
- 直接拼 query 可能导致检索词不够聚焦

先做轻量版本：

```text
目的地 + 偏好 + 备注关键词 + 场景词
```

例如：

```text
原始输入：
大理，自然风景、拍照、美食、轻松，不想太早起床，希望看日落

改写后的检索 query：
大理 轻松 日落 洱海 双廊 拍照 美食 行程 攻略 推荐
```

第一阶段不一定要用 LLM 改写，可以先做规则增强：

- 提取目的地
- 合并偏好
- 从备注中补关键词
- 加入旅行场景词：景点、行程、攻略、推荐

后续再考虑：

- LLM Query Rewrite
- 多 query 扩展
- 查询纠错和别名补全

面试表达：

> 我对 RAG 检索 query 做了轻量改写，把目的地、偏好和用户备注组合成更适合向量检索的表达，提升口语化输入下的召回命中率。

---

### 3.4 Rerank

目标：先粗召回，再重排序。

推荐做法：

1. Chroma 先召回 top 10
2. Rerank 选出 top 3 或 top 5
3. 再把最终片段注入模型上下文

第一阶段可以先不用复杂模型，先做轻量规则 rerank：

- 命中目的地关键词加分
- 命中偏好关键词加分
- 命中备注关键词加分
- 片段过短或明显无关降分

后续再考虑：

- bge-reranker
- 交叉编码器 rerank
- LLM 相关性打分

面试表达：

> 我计划在基础向量召回之后增加 rerank 层，先召回更多候选片段，再根据目的地、偏好和备注关键词进行重排序，减少无关上下文进入模型。

---

### 3.5 Context Compression

目标：减少冗余上下文，只保留对生成有用的信息。

当前问题：

- 召回片段可能过长
- 可能包含和用户当前需求无关的信息
- 过多上下文会浪费 token，也会干扰生成

轻量做法：

- 对召回片段做长度截断
- 优先保留包含目的地、景点、偏好词的句子
- 对多个片段做去重

后续做法：

- LLM 压缩
- Map-reduce 摘要
- 只抽取“景点、特色、适合人群、注意事项”

面试表达：

> RAG 不只是召回越多越好，还要控制上下文质量。我后续会做检索结果压缩，减少冗余内容，让模型更聚焦于用户当前需求。

## 4. 本阶段暂时不优先做什么

为了避免路线发散，本阶段先不优先做：

- 多 Agent 协同
- GraphRAG 完整实现
- 复杂 MCP 改造
- 大规模评测平台
- 模型微调
- 复杂部署流水线

这些不是不重要，而是现在优先级低于：

```text
看清当前 RAG
-> 建立小评估集
-> 优化 query
-> 再做 rerank / compression
```
