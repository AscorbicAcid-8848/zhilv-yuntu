# RAG 在线阶段优化笔记

> 这份文档用于整理 `智旅云图` 下一阶段 RAG 优化思路。  
> 当前优先优化的不是离线入库阶段，而是用户 query 进入系统后，到上下文送入 LLM 之前的 **在线检索阶段**。

---

## 1. 先区分离线阶段和在线阶段

### 离线阶段

离线阶段主要负责把原始知识准备好。

典型流程：

```text
原始文件
-> 解析与清洗
-> 结构化分块 chunking
-> 文本嵌入 embedding
-> 写入向量索引 Chroma
```

你当前项目里已经完成了基础版：

- 本地 Markdown 攻略
- 文本切片
- embedding 向量化
- Chroma 入库
- 基础向量检索

所以离线阶段现在不是最优先的优化点。

---

### 在线阶段

在线阶段发生在用户真正发起请求时。

典型流程：

```text
用户 Query
-> Query 改写 / 子问题分解 / HyDE
-> 检索：向量相似度 Top-K + 可选 BM25 混合
-> Cross-Encoder 重排序
-> MMR 去冗
-> 上下文压缩
-> 拼接 Prompt
-> LLM 生成
```

当前项目更适合优先优化这一段，因为：

- 用户输入通常比较口语化
- 当前 query 构造还比较简单
- 当前主要依赖基础向量相似度
- 还没有系统评估召回结果
- 还没有 rerank、compression、去冗等增强步骤

---

## 2. 当前优先优化的是 RAG 在线阶段

你的理解是对的：

> 当前优先优化的是 RAG 在线问答阶段，也就是从用户 query 到 LLM 上下文之间的链路。

这一段可以重点关注：

- Query Rewrite / Query Correction
- 混合检索
- Rerank
- MMR 去冗
- Context Compression
- Prompt 拼接质量

一句话总结：

> 我当前优先优化的是 RAG 在线检索阶段，重点是从用户 query 到 LLM 上下文之间的链路，包括 Query Rewrite、向量召回、后续混合检索、Rerank、去冗压缩和 Prompt 拼接质量优化。

---

## 3. 为什么先优化在线阶段

因为当前项目里：

- 离线阶段已经有基础闭环
- 在线阶段仍然比较基础
- 在线优化能直接影响生成质量
- 在线优化更容易通过调试脚本和评估集验证
- 面试时也更容易讲清楚“我不是只会接一个向量库”

相比马上做 GraphRAG 或多 Agent，先把在线检索链路打磨清楚更稳。

---

## 4. 几个核心优化点

### 4.1 Query Rewrite / Query Correction

解决的问题：

- 用户输入太口语化
- 偏好和备注比较散
- 原始 query 不够适合检索
- 城市别名、景点别名、模糊表达可能影响召回

例子：

```text
原始输入：
大理，自然风景、拍照、美食、轻松，不想太早起床，希望看日落

改写后的检索 query：
大理 轻松 日落 洱海 双廊 拍照 美食 行程 攻略 推荐
```

第一阶段建议先做轻量规则版：

- 保留目的地
- 合并用户偏好
- 从备注中提取关键词
- 补充旅行场景词：景点、行程、攻略、推荐

后续再考虑：

- LLM Query Rewrite
- Query Correction
- 子问题分解
- HyDE

---

### 4.2 混合检索

当前项目主要是向量检索。

后续可以考虑：

```text
向量检索 + BM25 关键词检索
```

两者分工：

- 向量检索擅长语义相似
- BM25 擅长关键词精确匹配

适合解决的问题：

- 用户明确提到某个景点名
- 关键词非常关键但语义相似度不一定高
- 向量检索召回结果过于泛化

注意：

混合检索会引入额外索引和融合逻辑，改动比 Query Rewrite 更大，所以不建议第一个做。

---

### 4.3 Rerank

基础向量检索返回的 top-k 不一定排序最优。

推荐流程：

```text
Chroma 先召回 top 10
-> Rerank 重新排序
-> 选择 top 3 / top 5
-> 注入模型上下文
```

第一阶段可以先做规则版 rerank：

- 命中目的地关键词加分
- 命中偏好关键词加分
- 命中备注关键词加分
- 命中评估集中 expected keywords 加分
- 明显无关片段降分

后续再考虑：

- Cross-Encoder
- bge-reranker
- LLM 相关性打分

---

### 4.4 MMR 去冗

MMR 主要解决：

- 召回片段之间过于相似
- 多个片段重复描述同一个景点
- 上下文信息不够多样

它希望在相关性和多样性之间做平衡。

在旅行规划里，MMR 可以避免：

- top-k 全是“大理古城”
- 忽略洱海、双廊、喜洲等其他相关地点

---

### 4.5 Context Compression

Context Compression 解决的问题：

- 召回片段太长
- 上下文冗余
- 无关内容进入 prompt
- token 浪费
- 模型注意力被干扰

第一阶段可以先做轻量版：

- 片段长度截断
- 只保留命中关键词附近句子
- 合并重复内容
- 去掉明显无关句子

后续可以考虑：

- LLM 压缩
- 摘要式压缩
- 结构化抽取：景点、特色、适合人群、注意事项

---

## 5. 推荐执行顺序

虽然 Query Rewrite、混合检索、Rerank、压缩都值得做，但不建议一次性全上。

更稳的顺序是：

```text
RAG 可观测
-> Query Rewrite
-> 轻量 Rerank
-> MMR / Context Compression
-> 混合检索
```

---

## 6. 第一步为什么不是直接 Query Rewrite

因为你需要先看清楚当前检索效果。

如果没有可观测能力，你会不知道：

- 当前 query 是什么
- top-k 召回了哪些片段
- 召回片段是否相关
- 优化后有没有变好

所以第一步应该先做：

```text
RAG 调试脚本
```

它的目标是输出：

- 原始 query
- 改写 query
- top-k 召回片段
- 来源文件
- 排序结果
- 命中关键词

这样你后续做 Query Rewrite、Rerank、Compression 时，才有依据。

---

## 7. 推荐新增的调试脚本

建议新增：

```text
backend/scripts/debug_rag_retrieval.py
```

功能：

- 输入目的地
- 输入偏好
- 输入备注
- 输入 top_k
- 打印当前 RAG 召回结果

示例输出：

```text
Original Query:
大理 自然风景 拍照 美食 轻松 不想太早起床，希望安排一个适合看日落的地点。

Rewritten Query:
大理 轻松 日落 洱海 双廊 拍照 美食 行程 攻略 推荐

Top 1:
source: data/dali_guide.md
content: ...

Top 2:
source: data/dali_guide.md
content: ...
```

## 8. 面试表达

如果面试官问：

**“你这个项目后续怎么优化 RAG？”**

可以这样回答：

> 我会优先优化 RAG 在线检索阶段。当前项目已经完成基础向量检索，下一步会先增加 RAG 可观测能力，输出原始 query、改写 query 和 top-k 召回片段；然后做轻量 Query Rewrite，把目的地、偏好和用户备注改写成更适合检索的表达；再进一步做 Rerank、MMR 去冗和上下文压缩，提升召回相关性，减少冗余上下文对模型生成的干扰。

如果面试官继续问：

**“为什么不先做 GraphRAG？”**

可以回答：

> GraphRAG 适合更复杂的实体关系建模，比如城市、景点、路线和主题标签之间的关系。但当前项目更需要先把基础 RAG 的在线检索质量打磨好，包括 query 改写、召回可观测、重排序和上下文压缩。等这些基础链路稳定后，再考虑图结构增强会更合理。

---

## 9. 当前结论

你的理解是对的：

- 当前先优化在线阶段
- 重点包括查询重写、混合检索、重排序、压缩去冗
- 但执行顺序上应该先做可观测

最终建议：

```text
先可观测
再 Query Rewrite
再 Rerank / 去冗压缩
最后混合检索
```
