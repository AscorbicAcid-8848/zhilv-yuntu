## 1. 技术栈原理速记

### 1.1 FastAPI 是什么

- 一个基于 Python 的现代 Web 框架
- 用来定义接口、接收请求、返回响应
- 相比 Flask，更强调类型提示、自动校验和自动生成 API 文档

你项目里主要用它做：

- 行程生成接口
- 编辑接口
- 保存 / 查询 / 删除接口
- 天气接口
- 导出接口

### 1.2 Pydantic 是什么

- 用来做数据校验和数据建模
- 你定义一个模型，传入的数据会自动按类型解析、校验和约束
- 在 FastAPI 里，经常用于请求体和响应体定义

你项目里主要用它做：

- `TripRequest`
- `TripEditRequest`
- `Itinerary`
- `DayPlan`
- `SpotItem`

### 1.3 SQLAlchemy 是什么

- 一个 Python ORM / 数据访问工具
- ORM 的意思是：用 Python 类操作数据库，而不是全手写 SQL
- 它不是数据库本身，而是“你怎么操作数据库”的那一层

项目里的关系：

```text
Python 代码 -> SQLAlchemy -> SQLite -> app.db
```

### 1.4 SQLite 是什么

- 一个轻量级关系型数据库
- 数据直接存成一个本地文件
- 适合个人项目、Demo、原型验证

它主要负责：

- 保存 itinerary
- 查询历史记录
- 删除历史数据

### 1.5 HTTPX 是什么

- 一个 Python HTTP 客户端库
- 用来在后端代码里请求外部 API

你项目里它用来：

- 调高德地图 Web 服务
- 调高德天气服务

### 1.6 LangChain 是什么

- 一个帮助你组织 LLM 应用流程的框架
- 它不是模型本身，而是“怎么调用模型、怎么接工具、怎么接上下文”的一层编排能力

你项目里它主要负责：

- 调用 `qwen-max`
- 生成结构化旅行草稿
- 单日编辑草稿生成

### 1.7 DashScope 是什么

- 阿里云提供的大模型服务平台
- 你当前项目通过兼容 OpenAI 接口的方式接入 `qwen-max`

### 1.8 RAG 是什么

- Retrieval-Augmented Generation，检索增强生成
- 先检索相关知识，再把知识片段作为上下文提供给大模型生成结果

你项目里的 RAG 流程：

```text
本地攻略文档
-> 文档切片
-> 向量化
-> ChromaDB 检索
-> 把结果注入 prompt
-> 大模型生成 itinerary
```

### 1.9 ChromaDB 是什么

- 一个向量数据库
- 用来存 embedding 向量，并根据相似度做检索

你项目里它主要负责：

- 保存攻略切片的向量
- 根据用户目的地和偏好召回相关攻略片段

### 1.10 MySQL、SQLite、Redis、Chroma 的区别是什么

可以先把它们分成 3 类理解：

```text
MySQL / SQLite = 关系型数据库
Redis = 内存型键值数据库
Chroma = 向量数据库
```

- `MySQL`
  - 适合正式线上系统
  - 更适合高并发、多人访问、长期演进
- `SQLite`
  - 更轻量
  - 适合个人项目、原型和本地存储
- `Redis`
  - 主要用来做缓存、会话、队列和热点数据存储
  - 重点不是替代数据库，而是提升速度
- `Chroma`
  - 主要解决语义检索问题
  - 用于 RAG 场景下的向量召回

在你的项目里可以这样讲：

- `SQLite` 负责正式保存旅行方案和历史记录
- `Chroma` 负责存攻略切片向量并做语义检索
- `Redis` 未来适合负责天气、地图和检索结果缓存

这样回答会很清楚地体现“多存储系统分工”。

### 1.11 Agent 是什么

- 广义上是“能根据目标调用模型和工具完成任务的智能体”
- 不一定非要多 Agent 才叫 Agent
- 单个 Agent + 工具调用 + 状态流编排，也可以算 Agent 应用

你这个项目里最接近 Agent 的部分是：

- `trip_planner_agent.py`
- `rag_tool.py`
- 后续可扩展成 LangGraph / MCP / 多 Agent

## 2. 后端与 AI 基础八股 QA

### Q1：FastAPI 和 Flask 有什么区别？

简答版：

> Flask 更轻量灵活，FastAPI 更偏现代 API 开发，内置类型校验和 Swagger 文档，对我这种需要大量请求体 / 响应体建模的项目更友好。

### Q2：Pydantic 和 dataclass 有什么区别？

简答版：

> dataclass 主要是简化类定义，Pydantic 更适合接口场景，因为它能自动校验和解析输入数据。

### Q3：ORM 是什么？为什么用 SQLAlchemy？

简答版：

> ORM 就是对象关系映射，用 Python 类映射数据库表。SQLAlchemy 可以让我用模型类和 session 操作数据库，开发效率更高，也更利于维护。

### Q4：SQLite 适合什么场景？有什么局限？

简答版：

> SQLite 很适合个人项目、原型和轻量场景，因为部署简单、无额外服务依赖。但它不适合高并发和复杂分布式场景，后续如果规模扩大可以迁移到 MySQL 或 PostgreSQL。

### Q5：为什么要用 HTTPX？

简答版：

> 因为项目需要在后端主动请求高德地图和天气等外部服务，HTTPX 的接口清晰，也支持同步和异步扩展。

### Q6：LangChain 在你项目里解决了什么问题？

简答版：

> 它主要帮助我把模型调用、结构化输出和工具上下文组织得更清晰。虽然直接调模型 API 也能做，但 LangChain 更利于后续扩展复杂工作流。

### Q7：什么是结构化输出？

简答版：

> 就是要求模型按固定 JSON 结构返回结果，而不是自由文本。这样后端更容易解析、校验和继续处理。

### Q8：为什么 LLM 输出需要校验？

简答版：

> 因为模型输出不稳定，可能字段缺失、格式错误或者内容不完整。校验可以保证接口结果稳定，也方便失败时回退到规则逻辑。

### Q9：RAG 的核心价值是什么？

简答版：

> 它让模型生成时不只依赖参数记忆，而是能结合外部知识片段生成更贴近业务场景的结果。

### Q10：向量数据库为什么能做检索？

简答版：

> 因为文本经过 embedding 后会变成向量，语义相近的内容在向量空间里距离也更近，所以可以通过相似度检索相关文档。

### Q11：Redis 在这种项目里有什么价值？

简答版：

> Redis 可以做缓存、热点目的地数据复用、会话态管理、验证码或异步任务状态保存，能提升响应速度并减轻数据库压力。

20 秒版：

> 我在项目里主要缓存了三类内容：第一是天气，因为同城天气短时间内变化不大；第二是地图查询，因为地理编码、POI 和路线结果重复度很高；第三是 RAG 检索结果，因为高频 query 会重复命中相似攻略片段。这样可以减少外部调用和重复检索开销，提升响应速度和稳定性。

10 秒版：

> 我主要缓存了天气、地图查询和 RAG 检索结果，因为这三类请求重复度高、结果相对稳定，适合用 Redis 做缓存层。

参考回答：

> 我在项目里引入 Redis 作为缓存层，主要缓存天气查询、高德地图的地理编码 / POI / 路线结果，以及 RAG 检索结果。因为这些请求存在明显重复，且结果在一定时间内相对稳定，所以通过 Redis 用空间换时间，减少外部 API 调用和重复检索开销，提升了系统响应速度和稳定性。

### Q12：LangGraph 和 LangChain 的区别？

简答版：

> LangChain 更适合快速搭建链路和工具调用，LangGraph 更适合把复杂流程拆成状态机 / 图结构，便于控制多步骤、多分支和可恢复流程。

### Q13：MCP 是什么？适合放在哪？

简答版：

> MCP 可以理解为一种标准化工具接入协议，适合把地图、天气、POI 这类外部能力抽成统一工具层。当前项目先用普通 service 直连 API 更简单，后续再 MCP 化更合理。

### Q14：你这个项目用了 function calling 吗？

简答版：

> 严格说还不算标准的模型原生 function calling。当前项目主要是后端显式编排：由服务层按既定流程调用 RAG、地图、天气等能力，模型负责生成结构化结果。

参考回答：

> 我这个项目里的工具调用主要发生在后端，由服务层按既定流程显式编排；而 function calling 更强调由模型先判断是否需要调用工具、调用哪个工具以及传什么参数，后端再负责实际执行和结果回传。

两种方式核心的区别：

你现在的项目：

```text
后端先决定流程
-> 查 RAG
-> 查地图
-> 查天气
-> 模型生成结果
```

function calling：

```text
用户提问
-> 模型判断要不要调用工具
-> 模型返回 tool/function call
-> 后端执行工具
-> 工具结果再回给模型
-> 模型继续回答
```

以下是根据你提供的图片内容转换的 Markdown 表格：

| 维度 | 你的现状（显式编排+结构化输出） | FunctionCalling（工具调用） |
|------|-------------------------------|----------------------------|
| 谁决定下一步干嘛？ | 你（开发者写的后端代码） | 大模型自己（动态推理） |
| 业务流程 | 静态、确定性强、像流水线 | 动态、不可预测、像自主思考 |
| JSON的用途 | 提取出想要的数据（Data Payload） | 发出执行某个动作的命令（Command） |
| 优点 | 极其稳定！100%按照你设计的流程走，好排错。 | 极其灵活！能处理复杂的、甚至你没提前预料到的多步用户需求。 |
| 缺点 | 用户一旦提出稍微偏离预设流程的问题，系统就傻眼了。 | 容易“翻车”（幻觉）。大模型可能会乱调工具、陷入死循环。 |

## 3. 后端深入 QA

### Q1：从 `POST /trip/generate` 到最终返回结果，后端数据流怎么走？

参考回答：

> 请求先进入 `trip.py` 路由层，由 Pydantic 完成请求体验证；然后路由调用 `trip_service.py` 作为主编排层。`trip_service.py` 会先收集 RAG 上下文，再调用 `trip_planner_agent.py` 生成结构化草稿；草稿生成后，再进入地图 enrich 流程，用高德服务补地址、坐标、路线和图片；最后统一刷新预算明细，组装成 `Itinerary` 响应模型返回前端。如果用户选择保存，再由 `storage_service.py` 通过 SQLAlchemy 写入 SQLite。

### Q2：为什么要把路由层和服务层分开？

参考回答：

> 路由层只负责接收请求、参数校验和返回响应，尽量保持薄；服务层负责真正的业务编排，比如 RAG 检索、模型调用、地图 enrich、预算刷新和持久化。这样分层后，接口逻辑更清晰，也更方便测试和后续扩展。

### Q3：为什么 `trip_service.py` 是核心，而不是直接在路由里写逻辑？

参考回答：

> 因为这个项目的主复杂度不在 HTTP 本身，而在“多模块协同的业务编排”。如果把生成、编辑、地图补全、预算计算和存储全部写在路由里，代码会非常混乱。把核心逻辑放在 `trip_service.py`，可以让路由只做入口，服务层负责真正的主流程控制。

### Q4：Pydantic 在这个项目里除了校验，还有什么价值？

参考回答：

> 除了基础校验，Pydantic 还有三个价值：第一是统一请求和响应结构，前后端联调更稳定；第二是约束 LLM 输出后的数据形态，避免结果乱掉；第三是让 FastAPI 自动生成清晰的接口文档。

### Q5：为什么生成结果之后还要做地图 enrich，而不是让模型直接生成？

参考回答：

> 因为模型适合做语义规划，比如“去哪儿、怎么安排”；但坐标、POI、路线距离和耗时这类信息需要高精度、可验证的数据来源，更适合交给地图服务。也就是说，模型负责“规划”，外部服务负责“校准和补全”。

### Q6：为什么预算不是完全交给模型算？

参考回答：

> 因为模型给预算容易模板化、波动大、不稳定。我现在的设计是：模型负责语义草稿，后端规则负责预算拆分和刷新，这样结果更可控，也更容易解释。后续如果要升级，也应该是“模型给建议、后端做校验与修正”，而不是完全相信模型。

### Q7：为什么要保存前端当前版本再导出 PDF？

参考回答：

> 因为页面上的最新结果可能经过天气提示修正、编辑或重新保存，如果直接导出数据库旧版本，会出现“页面和导出结果不一致”的问题。所以导出前先同步一次，可以保证用户看到的内容和导出的内容一致。

### Q8：为什么 SQLite 现在够用？什么时候要换数据库？

参考回答：

> 现在这个项目是个人项目和原型验证，SQLite 足够支撑保存、查询、删除和历史管理，部署也简单。但如果后续加入多用户、高并发、缓存、任务队列和更复杂的查询需求，就更适合迁移到 MySQL 或 PostgreSQL。

### Q9：如果后端继续工程化，你最先做什么？

参考回答：

> 我会先做三件事：第一是 Redis 缓存和热点数据复用；第二是异步任务，把长耗时生成和导出拆出去；第三是日志、监控和错误追踪。这样能先把系统从“可运行”提升到“可维护、可扩展”。

### Q10：为什么要给天气、地图和 RAG 加缓存？

参考回答：

- 天气缓存
  - 因为同城天气短时间内变化不大
- 地图缓存
  - 因为地点坐标、POI、路线结果高度重复
- RAG 缓存
  - 因为高频 query 会重复命中相似攻略片段

## 4. RAG 深入 QA

### Q1：你这个项目里的 RAG 具体是怎么做的？

参考回答：

> 我目前用的是本地攻略文档 + ChromaDB 的轻量级 RAG。流程是：先准备 Markdown 攻略文档，然后切片、向量化并写入 Chroma；用户发起请求时，根据目的地、偏好和备注做相似度检索，召回相关攻略片段，再把这些片段注入到模型生成流程里。

### Q2：为什么要切片？如果不切片会怎样？

参考回答：

> 因为原始文档太长，直接整篇送进模型会带来上下文浪费，也会降低检索精度。切片后，检索单元更细，能更精准地召回和目标问题相关的内容。如果不切片，召回结果会过粗，噪声也更大。

### Q3：切片大小怎么决定？

参考回答：

> 切片大小本质上是在“上下文完整性”和“检索精度”之间做平衡。切太小可能上下文断裂，切太大又容易召回一大段无关内容。通常要结合文档结构、段落长度和业务场景来调。旅行攻略这种文本，我更倾向于按主题段落或小节切，而不是机械按字数切。

### Q4：为什么需要向量化？

参考回答：

> 因为向量化能把文本映射到语义空间里，后续检索不再只靠关键词，而是靠语义相似度。这样即使用户表达和攻略原文不完全一致，也能找到相关片段。

### Q5：为什么选 Chroma？

参考回答：

> 因为 Chroma 对个人项目和原型验证很友好，接入简单，和 Python 生态结合方便，足以支撑当前规模的向量存储和检索。对我这个项目来说，它的价值在于快速落地，而不是追求特别复杂的分布式能力。

### Q6：RAG 相比纯模型生成，最大的提升是什么？

参考回答：

> 最大提升是让结果更贴近具体目的地和业务场景。纯模型更容易生成泛泛的建议，而 RAG 能把真实攻略信息带进来，让景点、建议和描述更具体。

### Q7：RAG 的主要问题会出在哪？

参考回答：

> 主要问题一般出在四个地方：第一，切片不合理；第二，召回内容不准；第三，召回太多噪声；第四，召回到了但模型没用好。所以 RAG 不只是“检索一下”，而是切片、召回、注入和生成都要一起调。

### Q8：如果后续继续优化 RAG，你会怎么做？

参考回答：

> 我会优先从三个方向优化：第一，改进切片策略，按景点主题和段落结构切；第二，引入 query rewrite 或 rerank 提升召回质量；第三，补一套评估集，比较纯生成、RAG 生成和地图 enrich 后结果的差异。

### Q9：为什么第一版 Rerank 先用规则，而不是直接上复杂模型？

参考回答：

> 因为第一版优化的目标不是一步做到最强，而是先做出一个可解释、可调试、低成本的 rerank 闭环。我当前已经通过调试脚本观察到基础向量检索存在排序不够理想的问题，比如“文档开头”这种低信息量片段会混进 top-k，而“行程参考”类片段虽然相关，但不一定总能排在更靠前的位置。所以我先设计了一版轻量规则：标题命中权重高于正文命中，因为标题更能概括片段主题；“文档开头”这类已知噪声做强惩罚；“行程”类标题做额外加权，因为它和旅行规划任务更贴近。这样做的目的是先低成本验证 rerank 是否有效，再决定后续要不要升级到 cross-encoder 或更复杂的相关性模型。

### Q10：你现在是怎么拆分 RAG 在线阶段优化的？

简答版：
> 我现在把 RAG 在线阶段拆成了三步：先在 `rag_tool.py` 做 Query Rewrite，把用户需求改写成更适合检索的 query；再在 `retriever.py` 做基础召回后的轻量 Rerank；最后用调试脚本观察 query、top-k、分数和排序原因。

参考回答：

> 我当前优先优化的是 RAG 的在线阶段，也就是从用户 query 进入，到检索上下文送进 LLM 之前这一段。具体拆分上，`rag_tool.py` 负责 Query Rewrite，把用户目的地、偏好、节奏和备注整理成更适合检索的 query；`retriever.py` 负责基础向量召回后的轻量 Rerank，对候选片段重新排序，把更贴近旅行规划目标的片段排前面；另外我还加了 `debug_rag_retrieval.py`，把检索 query、top-k 召回结果、`rerank_score` 和 `rerank_reasons` 打印出来，保证这条链路是可观测、可验证、可迭代的。

### Q11：RAG 和联网搜索的关系是什么？

参考回答：

> 本地 RAG 更适合解决静态知识，比如目的地攻略、景点特色、旅行建议；联网搜索更适合实时信息，比如营业状态、节假日提醒、临时天气变化。二者不是替代关系，而是互补关系。

### Q12：如果面试官问“你真的掌握 RAG 吗”，你怎么回答？

参考回答：

> 我会说我已经掌握了 RAG 的基本流程，包括文档切片、向量化、向量库检索、上下文注入和结果生成，也在项目里实际做过本地攻略检索增强。但如果是更深入的 query rewrite、rerank、评估体系和复杂检索策略，我目前是在继续学习和迭代中。

## 5. 项目本身高频 QA

### Q1：这个项目解决了什么问题？

参考回答：

> 它解决的是“用户给出旅行需求后，如何快速得到一个可展示、可编辑、可保存的结构化旅行方案”这个问题。重点不只是生成文本，而是把生成、地图、天气、历史管理和导出连起来。

### Q2：为什么这个项目不只是一个 LLM Demo？

参考回答：

> 因为我没有停留在“让模型输出一段文字”，而是做了完整的业务链路，包括结构化 itinerary、RAG 检索、地图 enrich、天气展示、历史管理和 PDF 导出。

### Q3：为什么 itinerary 不能完全让模型生成？

参考回答：

> 因为像经纬度、地址、POI、路线耗时这类地理数据要求高精度，不适合让模型凭空生成。我把语义规划交给模型，把地理信息交给高德服务做后处理补全。

### Q4：你的核心设计决策是什么？

参考回答：

> 我最核心的设计决策是把系统拆成“模型负责语义规划，地图服务负责结构化地理信息，存储层负责持久化，前端负责展示和交互”的分工。这样比把所有事都压给模型更稳定。

### Q5：为什么要做 RAG？

参考回答：

> 因为纯模型生成容易泛化、空泛或者不贴合具体目的地。加入本地攻略检索后，生成结果会更贴近真实景点和目的地信息。

### Q6：项目里最难的点是什么？

参考回答：

> 我觉得最难的是让系统输出“看起来像真实产品结果”，而不是演示性质的假数据。所以我做了结构化输出校验、地图 enrich、天气感知提示、导出前自动保存等处理。

### Q7：为什么天气提示要单独处理？

参考回答：

> 因为我发现如果页面天气是小雨，但提示里还写“注意防晒”，会显得很假。于是我在前端增加了天气感知逻辑，对提示做过滤和补充，让展示结果更合理。

### Q8：为什么导出前要自动保存？

参考回答：

> 因为前端页面上的 itinerary 可能和后端数据库里的版本不完全一致。如果直接导出数据库版本，用户会觉得导出的内容和当前页面不一致，所以我在导出前先做了一次保存同步。

### Q9：这个项目如果继续做，你最先优化什么？

参考回答：

> 我会优先做三件事：第一是 Redis 缓存和任务队列；第二是 LangGraph 化，把工作流状态组织得更清晰；第三是建立 itinerary 质量评估指标，提升生成结果的一致性和可控性。

### Q10：如果要支持真实线上用户，你会补什么？

参考回答：

> 我会补用户系统、多用户数据隔离、缓存、限流、监控告警、日志追踪、任务队列，以及更稳定的导出和地图图片处理机制。

## 7. 速答模板

### 5.1 一分钟项目介绍

> 我做了一个叫“智旅云图”的智能旅行规划系统。用户输入目的地、日期、预算和偏好后，后端会基于 LangChain 和 `qwen-max` 生成结构化 itinerary，并结合本地攻略做 RAG 检索增强。生成之后，我又通过高德地图 Web 服务补充地址、坐标、路线和图片信息。前端使用 Vue 3 做结果展示，支持地图、天气、预算、历史管理和 PDF 导出。这个项目对我来说最有代表性的一点是，它不是单点的模型调用，而是一条完整的 AI 应用产品链路。

### 5.2 三分钟项目介绍

> 这个项目叫“智旅云图”，定位是一个面向中文旅行场景的智能规划系统。  
> 后端我用的是 FastAPI，主要负责行程生成、编辑、保存、查询、删除、天气和导出这些接口。  
> AI 部分我用 LangChain 接 DashScope 的 `qwen-max` 做结构化 itinerary 生成，并支持对某一天行程做单独智能编辑。  
> 为了避免结果太空泛，我引入了本地攻略文档和 ChromaDB，做了一个轻量级 RAG，把相关攻略片段作为上下文注入到生成流程中。  
> 同时我没有让模型直接生成经纬度和路线，而是接入高德地图 Web 服务，在 itinerary 生成后再补地址、坐标、POI、路线和图片，这样准确性更高。  
> 前端我用 Vue 3 + Vite 实现了规划页、结果页和历史页，并接入高德地图 JavaScript API 展示景点点位，还加入了天气展示和导出功能。  
> 这个项目里我比较重视“产品完成度”，所以还专门处理了几个问题，比如模型输出校验、导出前自动保存、天气和旅行提示一致性、地图 enrich 这类真实体验问题。  
> 如果继续做，我会优先考虑 Redis 缓存、LangGraph 工作流、评估指标和更完整的用户系统。

### 5.3 如果面试官追问“你具体负责了什么？”

> 这个项目是我自己主导完成的。我负责了后端接口设计、Pydantic / SQLAlchemy 数据结构建模、LangChain 生成链路、RAG 检索、地图服务封装、天气接口、导出服务，以及前端页面联调和交互展示。
